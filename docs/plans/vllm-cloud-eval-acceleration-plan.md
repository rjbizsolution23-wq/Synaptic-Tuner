## Summary

Speed up post-training evaluation by separating the training runtime from the evaluation runtime and introducing a dedicated vLLM-backed cloud evaluation path. Training should remain on the stable Unsloth image. Evaluation should move to a separate vLLM image profile that starts an OpenAI-compatible vLLM server after training, runs the scenario suite at higher throughput, and then performs exact per-example dataset loss computation before syncing a complete experiment artifact set.

The core principle is:

- keep training stable
- make evaluation fast
- keep loss exact
- preserve canonical experiment tracking and agent-readable analysis outputs

Target user flow:

1. User runs a normal experiment config.
2. Training completes on Unsloth.
3. Post-training evaluation starts on a dedicated vLLM runtime.
4. vLLM serves the scenario/test suite with higher throughput.
5. The same job, or a tightly coupled second phase in the same runtime family, computes per-example dataset loss.
6. Results are saved under the canonical experiment structure and exposed through CLI/skills for agent analysis and next-run hypothesis generation.

This plan assumes Hugging Face Jobs is the first implementation target, but the design must remain provider-agnostic so Modal and RunPod can use the same contracts later.

#### Design Approach

We should explicitly split the evaluation stack into two runtime modes:

1. **Stable Mode**
- current direct Unsloth evaluation
- lowest risk
- slower

2. **Fast Mode**
- dedicated vLLM evaluation runtime
- higher throughput for scenario suite
- exact dataset loss computed after scenario eval using the same job workspace

The training/eval interface should remain provider-neutral:

- training artifacts stay in canonical `runs/...`
- eval backend becomes a runtime choice, not a provider choice
- experiment records and analysis bundles remain backend-agnostic

#### Core Principle: Hybrid Fast Eval, Exact Loss

Do not overload one engine with incompatible goals.

- vLLM is the throughput engine for prompt/case evaluation
- exact model-forward loss remains the quality-analysis engine

This yields:

- faster scenario throughput
- exact per-example loss values
- no need to distort tracking or analysis schemas

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Eval runtime | A) Unsloth only B) vLLM only C) Hybrid | **C) Hybrid** | Fast eval + exact loss |
| Image strategy | A) Patch Unsloth image B) Dedicated vLLM image | **B) Dedicated vLLM image** | Cleaner dependency isolation |
| Loss placement | A) Separate job B) Same job after eval | **B) Same job after eval** | Avoid extra model download/setup when possible |
| Throughput strategy | A) Bigger GPU only B) Concurrency only C) vLLM + concurrency + better presets | **C) vLLM + concurrency + better presets** | Best practical gain |
| Provider abstraction | A) HF-specific path B) runtime-agnostic contract | **B) Runtime-agnostic contract** | Modal/RunPod follow-on support |

### DAG / QDCL Specialist Plan

This plan assumes a DAG-shaped specialist-agent workflow. “QDCL” here means the orchestration should be explicit about:

- **Q**ueue: what work is ready now
- **D**ependencies: what must finish before downstream work starts
- **C**heckpoints: what artifacts/decisions each node must emit
- **L**earning loop: what evidence gets fed back into the next experiment

The important part is not the acronym itself. The important part is that each subagent owns a node with:

- explicit inputs
- explicit outputs
- disjoint file ownership where possible
- a defined downstream consumer

#### Specialist Operating Rules

Each specialist node should obey these constraints:

- it owns a narrow write set and should not revert unrelated edits
- it must emit a checkpoint artifact or decision note before downstream work starts
- it must record unresolved assumptions in a machine-readable handoff summary
- it should produce outputs that can be consumed by either a human or a follow-on agent without rereading the whole codebase

#### QDCL Control Surface

| Dimension | Requirement | Concrete Mechanism |
|-----------|-------------|--------------------|
| Queue | Know what can run now | Orchestrator marks nodes `ready`, `blocked`, `running`, `done` |
| Dependencies | Prevent speculative downstream edits | Each node declares upstream artifacts and exact blocking conditions |
| Checkpoints | Make progress resumable | Write plan notes, test outputs, or config decisions after each node |
| Learning Loop | Feed experiment evidence back into next runs | Persist runtime benchmarks, eval throughput, and artifact quality findings into experiment analysis outputs |

### Specialist DAG

```text
Node A: runtime-research specialist
  Output: pinned vLLM image/profile recommendation + startup contract
  ↓
Node B: eval-runtime specialist
  Depends on A
  Output: cloud_hf_job_vllm.py + server launch path
  ↓
Node C: evaluator-throughput specialist
  Depends on B
  Output: parallel evaluator execution with ordered progress artifacts
  ↓
Node D: experiment-orchestration specialist
  Depends on B
  Output: run-experiment/runtime routing and recovery wiring
  ↓
Node E: artifact-analysis specialist
  Depends on C and D
  Output: embedded loss + feature dataset + analysis bundle compatibility
  ↓
Node F: test-and-smoke specialist
  Depends on C, D, E
  Output: focused tests + one real HF smoke run
```

#### Node Contracts

| Node | Inputs | Outputs | Write Scope | Exit Checkpoint |
|------|--------|---------|-------------|-----------------|
| A. runtime-research | current HF eval path, official vLLM docs, HF GPU constraints | pinned image recommendation, server flags, unsupported-risk list | plan notes only | `runtime_decision.md` or plan amendment |
| B. eval-runtime | A outputs, existing `cloud_hf_job.py`, `vllm_setup.py` | `cloud_hf_job_vllm.py`, startup and readiness flow | `Evaluator/cloud_hf_job_vllm.py`, related tests | vLLM helper starts and reaches ready state locally/in tests |

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| vLLM image works differently across GPU flavors | High | Start with one pinned image and one supported flavor |
| Parallel eval breaks progress ordering or bucket sync | Medium | Keep single writer semantics and test partial artifact integrity |
| LoRA serving mismatch in vLLM | High | Start with one known-good LoRA model family and explicit smoke |
| Exact loss step diverges from current semantics | High | Reuse current `per_example_loss.py` implementation unchanged |
| Over-optimizing HF-specific logic | Medium | Keep runtime selection/provider selection separate in contracts |

## Scope Assessment

This is a focused performance plan, not a full cloud-runtime rewrite.

In scope:
- dedicated vLLM eval runtime
- runtime selection
- evaluator concurrency
- same-job exact loss after fast eval
- experiment orchestration wiring

Out of scope:
- replacing Unsloth training
- turning all providers over at once
- changing experiment-tracking schemas beyond what is needed for runtime metadata
