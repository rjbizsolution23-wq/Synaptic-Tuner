## Summary

Change the default post-training orchestration from:

1. train
2. eval
3. loss

to:

1. train
2. launch eval
3. launch exact loss
4. wait for both
5. analysis / recommendation

The current default path is operationally convenient but no longer the right default:

- eval and loss both depend only on the completed training artifact
- they do not depend on each other
- running them serially increases wall clock unnecessarily
- embedding loss inside eval muddies failure handling and stage accounting
- the best hardware for eval is not always the best hardware for exact loss
- benchmark comparisons become harder when eval and loss are coupled

The target state is:

- **parallel by default** for full experiment runs
- **independent stage lineage** for eval and loss
- **stage-specific hardware selection**
- **clean retries** when one stage fails and the other succeeds
- **same-job loss kept only as an explicit fallback/smoke mode**

This plan is specifically about orchestration, not changing the underlying evaluation or loss math.

#### Design Approach

Introduce an explicit post-training execution policy with two modes:

1. `parallel`
   - default for full experiment runs
   - train completes
   - eval and loss submit as separate jobs
   - analysis waits for both

2. `same_job`
   - optional fallback/smoke mode
   - eval job may also compute exact loss in the same remote job
   - loss stage then recovers from eval artifacts instead of submitting a second job

Proposed spec surface:

```yaml
post_training:
  mode: parallel   # parallel | same_job
```

If we want to avoid a new top-level section, an acceptable fallback is:

```yaml
evaluation:
  loss_mode: parallel   # parallel | same_job
```

but `post_training.mode` is cleaner because it describes orchestration, not just eval behavior.

#### Target Execution Model

```text
training
   ↓
submit evaluation ──────┐
                        ├── wait for both ──> analysis ──> recommendation
submit loss ────────────┘
```

This should be implemented as true orchestration concurrency, not “submit one, poll to completion, then submit the other.”

#### Stage Contracts

**Evaluation**
- owns only evaluation outputs
- should not assume responsibility for exact loss unless `same_job` mode is explicitly enabled
- writes:
  - `evaluation_results.json`
  - `evaluation_results.md`
  - `evaluation_lineage.json`
  - progress / stage logs

**Loss**
- owns only exact loss outputs
- writes:
  - `per_example_losses.jsonl`
  - `loss_summary.json`
  - `loss_lineage.json`
  - shard/manifests/progress logs

**Analysis barrier**
- should start only when:
  - training is complete
  - eval is complete or explicitly skipped
  - loss is complete or explicitly skipped

#### Recovery Model

The orchestrator should support these states cleanly:

- eval completed, loss running
- eval failed, loss completed
- eval completed, loss failed
- eval running, loss failed
- eval failed, loss not submitted yet

Rules:
- completed stages are never resubmitted unless explicitly requested
- failed eval and failed loss remain retryable on resume
- analysis does not run until all selected post-training stages are terminal
- lineage/status should reflect that eval and loss are siblings derived from training

#### Hardware Model

Parallel orchestration is especially useful because hardware can differ by stage:

- training GPU: chosen for adapter training throughput / fit
- eval GPU: chosen for generation throughput
- loss GPU: chosen for exact forward-pass throughput

This is already conceptually present in the spec, but the current sequential embedding blurs it.

#### Phase 2: Eval/ Loss Decoupling

Remove the implicit same-job coupling from the default eval path.

Work:
- stop forcing `with_loss=bool(spec.loss.enabled)` in the eval runner for the default mode
- only pass same-job loss flags when `post_training.mode == "same_job"`
- keep current same-job recovery path intact for fallback mode

Deliverable:
- eval job is evaluation-only by default

#### Phase 3: Parallel Orchestrator

Teach the experiment orchestrator to launch eval and loss as sibling post-training jobs.

Work:
- submit eval and loss after completed training
- avoid polling one to completion before submitting the other
- introduce a small concurrent wait/join layer
- keep stage tracking updates independent

Implementation note:
- this can be done with simple threaded futures or a small coordinator abstraction; avoid overengineering

Deliverable:
- eval and loss run in parallel after training

#### Phase 4: Analysis Barrier And Retry Semantics

Make analysis/recommendation wait correctly and recover correctly.

Work:
- only run analysis when all selected post-training stages are terminal
- preserve existing retry semantics for failed eval/loss
- ensure partial success states are represented cleanly in experiment status

Deliverable:
- stable resume / retry behavior

#### Phase 5: Tests, Docs, And Operator UX

Work:
- update canonical skill guidance
- document that `parallel` is default and `same_job` is fallback
- add test coverage for:
  - default parallel mode
  - explicit same-job mode
  - one stage succeeds / one fails
  - resume with one completed sibling stage

Deliverable:
- reliable operator workflow and stable documentation

## Risks And Mitigations

### Risk: Duplicate or colliding stage submissions
- **Issue**: concurrent submit paths can collide on prefixes or stale stage metadata
- **Mitigation**: reuse the unique-ID work already added for run/eval prefixes and keep stage-specific roots distinct

### Risk: Analysis starts too early
- **Issue**: existing flow assumes sequential stage completion
- **Mitigation**: add an explicit post-training join barrier in the orchestrator

### Risk: Same-job loss fallback regresses
- **Issue**: current users may still want the cheap embedded path for smoke runs
- **Mitigation**: keep `same_job` supported and tested, but make it explicit

### Risk: Job accounting becomes harder
- **Issue**: two sibling post-training jobs may complicate experiment status
- **Mitigation**: rely on stage lineage and stage status as source of truth, not a single aggregate job ref

### Risk: Provider-specific assumptions leak into orchestrator
- **Issue**: this is currently HF-focused
- **Mitigation**: keep the orchestration policy provider-agnostic; only the handlers stay provider-specific

## Recommended Phase Cut

Recommended first implementation slice:

1. add `post_training.mode` with default `parallel`
2. make eval default to evaluation-only
3. launch eval and loss as separate sibling jobs
4. gate analysis on both stage completions
5. keep same-job loss as explicit fallback mode

That is the smallest slice that delivers the main throughput win without redesigning the rest of the experiment system.

## Definition Of Done

This plan is done when:

- `run-experiment` defaults to parallel post-training eval and loss
- eval and loss submit as distinct sibling jobs after training
- each stage writes its own canonical artifacts and lineage without assuming the other
- analysis waits for both selected post-training stages
- failed eval/loss stages remain independently retryable on resume
- explicit `same_job` mode still works for smoke/fallback usage
- test coverage exists for the new orchestration and recovery behavior
- canonical skill/docs explain the new default and the fallback mode clearly
