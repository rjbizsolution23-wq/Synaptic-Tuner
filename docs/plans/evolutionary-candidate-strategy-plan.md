## Summary

Upgrade the evolutionary fine-tuning loop from a basic random-gradient-noise prototype into a usable search policy for SFT.

The current system is mechanically working:

- candidate generation is happening
- fitness evaluation is happening
- structured evolutionary events are being logged
- selection is happening at the configured optimizer-step cadence

But the current strategy is not yet competitive:

- all recent full-run selection events kept the baseline gradient
- the current `gradient_noise` variants are only marginally different from baseline
- the configured `min_improvement=0.01` is much larger than the actual observed fitness deltas
- the search distribution is isotropic Gaussian noise in the full gradient space, which is a weak default for LoRA SFT

The target state is:

1. lower-variance candidate generation
2. guided exploration around the baseline gradient
3. acceptance logic calibrated to real observed fitness deltas
4. stronger telemetry so we can tell whether a strategy is actually helping
5. experiment specs that make the next evolutionary benchmark easy to run in the cloud

This plan is specifically about candidate search quality and evolutionary decision policy, not about general cloud orchestration.

#### Design Approach

Implement three strategy tiers.

**Tier 1: Better default random search**

- keep current `gradient_noise`
- add optional noise annealing
- add optional mirrored / antithetic candidate pairs
- lower or rework acceptance threshold

This is the fastest path to an actual improvement.

**Tier 2: Guided low-rank search**

- create `guided_subspace`
- build a search subspace from recent accepted baseline gradients
- sample noise preferentially in that subspace plus a smaller orthogonal component
- keep baseline candidate as candidate `0`

This is the best fit to the current architecture and the most promising “novel but practical” strategy.

**Tier 3: SAM-like directional candidate**

- not a full replacement for evolutionary search
- add a candidate type inspired by sharpness-aware perturbation
- treat it as another deterministic candidate in the pool

This is a later phase because it is a different conceptual family from random search, but it can live naturally in the same candidate interface.

#### Acceptance Model

Current:

- absolute `min_fitness_improvement`

Target:

Support both:

- absolute threshold
- relative threshold

Proposed config:

```yaml
selection:
  method: best
  min_improvement: 0.0
  min_relative_improvement: 0.0001
  noise_floor_epsilon: 1.0e-6
```

Decision rule:

- if `best - baseline < max(min_improvement, baseline * min_relative_improvement, noise_floor_epsilon)`
  - keep baseline
- else
  - accept the best candidate

This fits the observed deltas much better than a fixed `0.01`.

#### Strategy Surface

Target strategy names:

- `gradient_noise`
- `antithetic_noise`
- `scale_variation`
- `combined`
- `guided_subspace`

Proposed config shape:

```yaml
evolutionary:
  enabled: true
  candidates: 6
  eval_frequency: 10
  warmup_steps: 10
  strategy:
    type: guided_subspace
    params:
      noise_scale: 0.03
      orthogonal_noise_scale: 0.01
      subspace_dim: 4
      history_size: 16
      mirrored: true
      anneal: cosine
      max_grad_norm: 1.0
  selection:
    method: best
    min_improvement: 0.0
    min_relative_improvement: 0.0001
```

#### Telemetry Model

The current `candidate_selected` event is good, but it should also record:

- `acceptance_margin`
- `required_margin`
- `accepted_nonbaseline`
- `strategy_name`
- `guide_norm`
- `subspace_dim_effective`
- `noise_scale_effective`
- `orthogonal_noise_scale_effective`
- `mirrored_pairs_used`

This makes it easy to answer:

- are candidates being accepted?
- which strategy is actually winning?
- is the guide subspace collapsing?

## Validation Strategy

#### Unit Tests
- candidate-count and pairing invariants for `antithetic_noise`
- guided-subspace basis construction and sampling invariants
- relative acceptance threshold behavior
- annealed noise schedule behavior
- event-payload field coverage

#### Integration Tests
- short local evolutionary smoke with:
  - `warmup_steps < max_steps`
  - known accepted nonbaseline case
- lineage summary contains new acceptance metrics
- cloud spec parsing and command construction carry new evolutionary fields

#### Runtime Verification

For a successful run we want to see, in `logs/evolutionary_events.jsonl`:

- some nonzero `accepted_nonbaseline`
- realistic margins, not all clipped by threshold
- stable effective noise scale
- no collapse to obviously random winners

Success is not “every event chooses a noisy candidate.”
Success is:

- candidates occasionally beat baseline
- acceptance policy is calibrated
- final model quality is at least competitive with baseline

## Risks

- guided-subspace state may add too much memory or CPU overhead if implemented naively
- acceptance thresholds that are too loose may accept noise too often and destabilize training
- comparing against earlier underpacked baselines can produce misleading conclusions
- adding too many strategies at once will make attribution weak

Mitigations:
- keep Phase 1 and Phase 2 small
- run a smoke before the full benchmark
- compare only against packed A100 baselines
