## Summary

Build a self-reinforcing fine-tuning loop on top of the existing ML pipeline: a lightweight model-serving proxy captures every inference as a structured log, an automated pipeline cleans and tags those logs as SFT/KTO/GRPO training candidates, versioned datasets accumulate across cycles, and users trigger retraining with a single CLI command. The architecture is deliberately lightweight — no new infrastructure services (Elasticsearch, MongoDB, Redis) — building almost entirely on components already in the repo (`shared/validation/`, `shared/judge/`, `Evaluator/`, `Trainers/`). The new components are a FastAPI logging proxy, a `shared/flywheel/` pipeline package, a SQLite catalog, and a `tuner flywheel` CLI subcommand.

#### Design Approach

The flywheel is a **linear data pipeline** with a feedback loop:

```
[User App / Client]
        │ OpenAI API (port 8080)
        ▼
[Logging Proxy]  ──logs──▶  inference_logs/YYYY-MM-DD.jsonl
        │ forward
        ▼
[vLLM :8000]  (GPU)
        │
        ▼
[File Watcher / Manual CLI]  (detects new logs)
        │
        ▼
[Ingestion Pipeline]  ──▶  canonical rollout format
        │
        ▼
[Data Cleaner]  (FitnessEvaluator + PII scrub)
        │
        ▼
[Auto-Tagger]  (rules → judge for ambiguous)
   ├──▶  Datasets/flywheel/vN/sft_training.jsonl
   ├──▶  Datasets/flywheel/vN/kto_training.jsonl
   └──▶  Datasets/flywheel/vN/grpo_training.jsonl  ⚠️ Handled during PREPARE
        │
        ▼
[Dataset Versioner]  (SQLite catalog + manifest.json)
        │
        ▼ "847 new records staged. Launch retrain?"
[Retrain Orchestrator]  ──▶  Trainers/sft/ or Trainers/kto/ or cloud
        │
        ▼
[New Model Artifacts]
        │
        ▼
[vLLM hot-swap or restart]  ──▶  loop closes
```

**Patterns applied:**
- **Decorator** on `BaseLLMClient.chat()` for zero-coupling inference logging
- **Strategy** for tagger (rule-based vs. LLM-as-judge, swappable)
- **Repository / Abstract Backend** for `LogCatalog` — async interface, SQLite local impl, Postgres cloud impl (see below)
- **Pipeline/Chain** for accumulate → tag → version → stage
- **Observer** for readiness heuristics (watches staged data growth)
- **GPU Mutex**: Serving and training cannot coexist on a single RTX 3090. Job orchestrator owns the lifecycle: stop vLLM → train → restart vLLM with new model.

#### Storage Backend Abstraction (Local vs. Cloud)

The flywheel runs in two distinct environments that require different storage backends. **The interface must be defined once and be environment-agnostic.** This mirrors how `ExperimentTracker` already works in this codebase (`create_tracker(backend="local"|"mlflow")`).

| Layer | Local (single-user, this repo) | Cloud (hosted platform, multi-tenant) |
|-------|-------------------------------|---------------------------------------|
| **Inference log catalog** | `aiosqlite` (`.tracking/flywheel.db`) | Postgres (per-tenant schema isolation) |
| **Dataset version manifests** | JSONL files in `Datasets/flywheel/` | Object storage (S3/R2/GCS) + Postgres metadata |
| **RunRegistry** | `.tracking/registry.jsonl` (PR #64) | Postgres table or per-tenant JSONL in object storage |
| **Actual training data** | JSONL in `Datasets/flywheel/vN/` | Object storage, downloaded at training time |
| **vLLM model artifacts** | Local filesystem | Object storage + vLLM `--lora-modules` path |

**`LogCatalog` abstract interface** (`shared/flywheel/catalog.py`):

```python
class LogCatalog(Protocol):
    async def insert_log(self, record: InferenceLogRecord) -> str: ...
    async def find_logs(self, filters: LogFilter) -> list[InferenceLogRecord]: ...
    async def mark_used(self, log_ids: list[str], dataset_version: str) -> None: ...
    async def create_dataset_version(self, ...) -> DatasetVersion: ...
    async def get_dataset_version(self, version_id: str) -> DatasetVersion | None: ...

# Implementations:
class SQLiteLogCatalog(LogCatalog): ...   # aiosqlite, local-first, zero-config
class PostgresLogCatalog(LogCatalog): ... # asyncpg, multi-tenant, connection pool

def create_catalog(backend: str = "sqlite", **kwargs) -> LogCatalog:
    # backend="sqlite" → SQLiteLogCatalog (default, local)
    # backend="postgres" → PostgresLogCatalog (cloud)
```

**Why async interface from day one:** SQLite sync works locally, but Postgres in a multi-tenant serving context requires async (`asyncpg`) to avoid blocking the inference proxy under concurrent load. Retrofitting sync → async later is a painful interface change. Using `aiosqlite` for the local backend keeps both implementations behind the same async contract at minimal extra cost.

**Config-driven backend selection:**
```yaml
# configs/flywheel/default.yaml (local)
catalog:
  backend: sqlite
  path: .tracking/flywheel.db

# configs/flywheel/cloud.yaml (platform)
catalog:
  backend: postgres
  url: ${DATABASE_URL}
  tenant_id: ${TENANT_ID}
  pool_size: 10
```

**Tenant isolation strategy (cloud):** Each tenant gets their own Postgres schema (`tenant_{id}.inference_logs`, `tenant_{id}.dataset_versions`). Schema-level isolation means a single Postgres instance serves all tenants with zero cross-tenant data leakage — no row-level filtering needed, no risk of a missing WHERE clause exposing another tenant's data.

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Log catalog interface | Sync, Async | **Async from day one** | Postgres requires async; retrofitting later is painful. `aiosqlite` for local, `asyncpg` for cloud — same interface. |
| Local log store | SQLite, DuckDB, flat JSONL | **SQLite via aiosqlite** | Zero config, ACID, async-compatible, Python stdlib. JSONL files stay as data layer for trainer compatibility. |
| Cloud log store | Postgres, MySQL, DynamoDB | **Postgres** | Schema-level tenant isolation, asyncpg performance, same SQL interface as SQLite layer. |
| Cloud data storage | S3, R2, GCS | **Configurable (S3-compatible)** | Use S3-compatible interface so R2/GCS/MinIO are all supported without code changes. |
| Tagger strategy | Rules only, LLM only, Hybrid | **Hybrid** | Rules handle clear pass/fail (~80%), LLM judge for ambiguous middle tier. Reuses existing `JudgeService`. |
| Retrain trigger | Automatic, cron, manual+readiness | **User-configurable** (auto/staged/manual, default staged) | Tenant controls policy. Readiness heuristics surface recommendation in TUI/API. |
| Model serving | vLLM, SGLang, Ollama | **vLLM** | Non-blocking LoRA hot-swap, stable tool-calling, OpenAI-compatible. Re-evaluate SGLang Q3 2026. |
| Inference logging | In-process wrapper, proxy | **FastAPI reverse proxy** | Transparent to caller, stable across vLLM versions, captures full request context. |
| Dataset versioning | DVC/lakeFS, append-and-snapshot | **Append-and-snapshot + manifests** | Immutable per-version directories, manifest.json for lineage. JSONL-compatible with trainers. |
| Multi-tenancy | Yes now, deferred | **Schema reserved** (`tenant_id` field in all records) | Single-user for v1. Postgres schema isolation design ready for cloud deployment. |
| GRPO reward signal | Environment execution, schema validation | **Schema validation in v1** | Tool call structural correctness (function name, arg schema) is a real reward. Full environment execution is v2. |
| Training: local vs cloud | Local only, both | **Both** | GPU mutex for local; cloud offload (HF Jobs/RunPod/Modal) for zero-downtime. Existing cloud backends used as-is. |

**Inference Log Record** (written by proxy to JSONL):
```json

  "schema_version": "1.0",
  "record_type": "inference_log",
  "timestamp": "ISO-8601",
  "model_id": "org/model-v2.1",
  "model_version": "training_run_id_or_commit_hash",
  "user_id": null,
  "request": {
    "messages": [{"role": "string", "content": "string"}],
    "temperature": 0.7,
    "max_tokens": 1024,
    "tools": []
  },
  "response": {
    "content": "string",
    "tool_calls": [],
    "finish_reason": "stop",
```


#### Integration with PR #64 — Unified Experiment Tracking

> **PR #64 (merged 2026-03-20)** added `shared/experiment_tracking/` with `RunRecord`, `RunRegistry`, `RunFilter`, adapters for all trainer types, and `LocalTracker` auto-registration. This substantially reduces the flywheel's registry work.

**Revised storage architecture:**

```
RunRegistry (.tracking/registry.jsonl)   ← PR #64 — tracks all training + eval + flywheel runs
    + links.jsonl                        ← lineage: inference_batch → flywheel_cycle → training → eval

SQLite catalog (flywheel-new, reduced scope)  ← inference log index ONLY
    inference_logs table                 ← fast tag/score/date filtering for dataset building
    dataset_versions table               ← manifest snapshots per retrain cycle

JSONL files (existing pattern)           ← actual conversation data
    inference_logs/YYYY-MM-DD.jsonl
    Datasets/flywheel/vN/sft_training.jsonl
```

**Flywheel adds a new run_type chain to RunRegistry:**
- `"inference_log_batch"` — batch of ingested + cleaned inference logs
- `"flywheel_cycle"` — tagging + staging run (tags: `dataset_version`, `record_count`, `sft_count`, `kto_count`, `grpo_count`)
- Links to existing `"sft"` / `"kto"` / `"grpo"` training runs via `registry.link_runs()`
- `dataset_source` field on RunRecord IS the dataset version identifier — no new field needed

**What this eliminates from original plan:**
- `shared/flywheel/registry.py` no longer needs to track training runs or model versions — RunRegistry already does that. Scope reduced to SQLite inference log catalog only.
- `shared/flywheel/stager.py` calls `RunRegistry.register_run()` + `link_runs()` directly instead of maintaining its own training run records.

## Cross-Cutting Concerns

| Concern | Status | Notes |
|---------|--------|-------|
| Security | Needs attention | Opt-in logging, tenant data isolation by design (platform owner cannot access tenant data). PII/PHI scrubber is a future feature. Tenant-held encryption keys. |
| Performance | Needs attention | Logging proxy must add <10ms overhead. Async queue pattern for non-blocking writes. Benchmark required in PREPARE. |
| Accessibility | N/A | CLI-only for v1 |
| Observability | Needs attention | Proxy exposes `/health` + `/metrics`. Job orchestrator writes structured JSON to `jobs/` directory. `tuner flywheel status` aggregates all signals. |
| GPU Resource Management | Needs attention | Single RTX 3090 cannot serve + train simultaneously. GPU mutex owned by orchestrator. Recommend cloud offload path as zero-downtime alternative. |
| Data Quality / Flywheel Poisoning | Needs attention | Auto-tagged data could degrade model over successive retrains. Mitigations: human review gate (first N cycles), holdout eval set, eval score tracking per model version. |
| Pre-commit hook | Needs attention | `print\s*\(.*token` false positive. All new code must use variable names that avoid "token" (use `api_key`, `hf_credential`, `auth_key`). |

## Open Questions

### Resolved Decisions

- [x] **GRPO in v1**: **Included.** Tool call schema validation provides the reward signal: +1 (valid tool call, correct function, args pass schema), +0.5 (correct function, partial arg failure), -0.5 (wrong function/missing required args), -1 (no tool call when expected). Environment-backed reward is a v2 enhancement.

- [x] **PII/PHI handling**: **Deferred as future feature.** Logging is opt-in per tenant. Strict tenant data isolation enforced at platform level — platform owner and other tenants cannot access any tenant's data, models, or logs. Tenant data encrypted at rest with tenant-held keys. PHI-grade scrubber (HIPAA-aware, conservative flag-for-review rather than auto-delete) is a future feature.

- [x] **Human approval gate**: **User-configurable policy.** Tenants select: fully automatic, staged (auto-prepare + manual launch), or fully manual. Platform default: staged. Policy stored per-tenant in flywheel config.

- [x] **Model serving**: **vLLM** with platform-owned API gateway abstraction. Non-blocking LoRA hot-swap (`load_inplace`), filesystem resolver for adapter auto-discovery, stable tool-calling across 20+ model families. Gateway layer owns auth, logging, routing, LoRA lifecycle — backend is swappable. Re-evaluate SGLang in Q3 2026 (async loading issue #8162, filesystem resolver PR #15454 may close the gap; SGLang offers 29% higher throughput + RadixAttention structured output advantage).

### Require Further Research (PREPARE phase)

- [ ] **Inference volume estimate**: How many inferences per day in typical use? Determines log rotation period and SQLite index strategy.
- [ ] **Target model size**: 3B/7B vs. 13B/14B affects vLLM memory tuning and training time estimates.
- [ ] **FitnessEvaluator score distribution on inference logs**: Are 0.8/0.3 thresholds reasonable starting points? Needs empirical calibration on a sample dataset.
- [ ] **vLLM LoRA hot-swap stability**: Needs testing with Qwen/Mistral architectures before committing to vLLM for v2.
- [ ] **Arena Learning applicability**: Microsoft's automated preference pair generation via "battles" between model versions could replace manual KTO labeling in v2.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Flywheel amplifies model biases/errors | Med | High | Holdout eval set never enters training; eval score comparison per model version; human review gate for first 3 cycles |
| GPU mutex causes serving downtime during training | High | Med | Clearly communicated downtime window; cloud offload option (HF Jobs/RunPod) eliminates downtime |
| PII leaks into training data | Low-Med | High | PII scrubber step in cleaner; opt-in logging policy; no logging of sensitive fields by default |
