## Summary

Automate the experiment data pipeline so every run emits consistent, ML-friendly metadata at the stage level, experiment level, and cross-run level without manual joins.

The current state is good but still uneven:

- `training_lineage.json` is the richest stage artifact
- `evaluation_lineage.json` captures quality outcomes, but not enough cost/runtime/planner context
- exact loss currently writes `loss_summary.json` and partial shard data, but it still lacks a canonical final lineage artifact named `loss_lineage.json`
- the experiment bundle can summarize runs, but still has to infer too much from mixed sources
- the benchmark ledger is useful, but it should be derived from stronger stage-level facts

The target state is a layered metadata system:

1. **Stage lineage as source of truth**
   - `training_lineage.json`
   - `evaluation_lineage.json`
   - `loss_lineage.json`

2. **Experiment-level joined bundle**
   - normalized rollup of training, eval, loss, recommendation, and analysis metadata

3. **Cross-run benchmark ledger**
   - append-or-upsert table for model/hardware/cost/speed comparisons

4. **ML-ready feature tables**
   - stable inputs for LightGBM and experiment search without ad hoc backfilling

The principle is:
- write facts once, as close to the producing stage as possible
- derive aggregates automatically
- keep all cost and runtime metadata tied to the stage that actually incurred it

#### Design Approach

The automation stack should use four layers.

**Layer 1: Stage lineage**

Each stage writes one canonical lineage document:

- `training_lineage.json`
- `evaluation_lineage.json`
- `loss_lineage.json`

Every stage lineage should include:

- identification
  - `stage`
  - `method`
  - `run_id`
  - `experiment_id`
  - `model_name`
  - `dataset.source`
  - `dataset.variant`
  - dataset fingerprint/hash when available
- hardware
  - requested flavor
  - resolved flavor
  - visible GPU count
  - tensor parallel / worker count if relevant
  - runtime/backend (`unsloth`, `vllm`, `transformers`, etc.)
- planner context
  - whether auto-hardware was used
  - optimization objective (`cost`, `balanced`, `speed`)
  - resolved batch / grad accumulation / effective batch
  - planner notes or fallback reason
- runtime
  - `started_at`
  - `finished_at`
  - `duration_seconds`
  - structured status
- pricing
  - `price_source`
  - `price_hour_usd`
  - `estimated_cost_usd`
- results
  - stage-specific metrics

**Layer 2: Stage partials**

Stages that stream data while running should also write partial metadata:

- training: latest progress JSONL plus structured summary
- evaluation: event stream, progress, partial counts
- loss: shard manifests, partial summary, top-loss partials

The final lineage document should point to the partial artifacts used to construct it.

**Layer 3: Experiment bundle**

The experiment bundle should join the stage lineage artifacts and produce:

- `feature_dataset.csv` / `feature_dataset.jsonl`
- `failure_slices/`
- `next_run_candidates.json`
- `draft_next_spec.yaml`
- benchmark ledger row upsert

This layer should not recompute stage facts if the lineage artifacts already contain them.

**Layer 4: Cross-run benchmark ledger**

The benchmark ledger should remain a derived comparison table:

- one row per experiment or benchmark branch
- populated from enriched stage lineage
- enough fields to answer:
  - which hardware is fastest?
  - which hardware is cheapest?
  - which cost/speed tradeoff is best for a given model size and method?

#### Canonical Lineage Contracts

**Training lineage**
- keep current rich training metrics
- normalize envelope fields so local and cloud runs match
- include:
  - dataset row count
  - planner objective and resolved hardware decision
  - live hourly price when known
  - final cost estimate
  - peak reserved VRAM, headroom, OOM risk

**Evaluation lineage**
- extend current quality summary with:
  - backend/runtime
  - requested vs effective GPU topology
  - runtime seconds
  - live hourly price and cost estimate
  - request error counts
  - fallback metadata such as “x4 requested, fell back to single GPU eval”

**Loss lineage**
- new canonical artifact
- `loss_summary.json` remains a supporting summary artifact, not the source-of-truth lineage filename
- should include:
  - `rows_written`
  - `examples_total`
  - `worker_count`
  - `batch_count`
  - `batch_max_tokens`
  - `adaptive_batching`
  - `completion_only`
  - `mean_loss`
  - `median_loss`
  - `p95_loss`
  - `max_loss`
  - `total_tokens`
  - `completion_tokens`
  - runtime/pricing/hardware fields matching the other stages
  - links to:
    - `per_example_losses.jsonl`
    - `high_loss_examples.jsonl`
    - manifests/shards

#### LightGBM / Search Readiness

The experiment bundle should emit stable features for downstream ML:

- model family / size bucket
- method
- dataset variant / row count / prune strategy
- stage hardware choices
- resolved training shape
- runtime and cost by stage
- eval summary by tag family
- loss distribution summary
- fallback / degraded / stalled markers

This turns experiment analysis into a structured data product instead of a report-only workflow.

## Risks

### Risk: Schema sprawl
- If each stage invents its own metadata names, automation gets harder instead of easier.
- Mitigation: define one normalized lineage field vocabulary up front.

### Risk: False precision in pricing
- If live HF pricing lookup fails, stale local pricing could silently mislead the ledger.
- Mitigation: write explicit blanks or `price_source=unavailable`, never guessed numbers.

### Risk: Backward compatibility breakage
- Older experiments do not have `loss_lineage.json` or enriched eval/training lineage.
- Mitigation: keep tolerant readers and additive fields only.

### Risk: Double sources of truth
- If experiment bundles and ledgers keep re-deriving different facts, data quality will drift.
- Mitigation: make stage lineage primary and push derivation downstream only.
