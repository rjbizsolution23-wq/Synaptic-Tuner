## Summary

Replace the current noisy, stage-specific cloud logs with a canonical structured health stream that is easy for operators and agents to inspect while jobs are running.

The immediate problem is operational:

- raw HF job stdout is dominated by pip, clone, tqdm, and server banners
- each stage emits a different log shape
- progress and health signals are mixed with debug noise
- `cloud-jobs` can show raw logs, but not a concise stage health summary
- when a stage fails, operators still have to manually interpret unrelated bootstrap output

The target is a three-layer logging model:

1. **Structured event stream**
   - append-only JSONL
   - compact lifecycle, progress, resource, sync, and failure events
   - consistent across training, eval, vLLM eval, and loss

2. **Rolled-up stage summary**
   - latest known state for fast inspection
   - one JSON file per stage
   - designed for `cloud-jobs` and future dashboards

3. **Raw debug log**
   - full stdout/stderr preserved separately
   - only used when the structured stream is insufficient

This plan is provider-agnostic. HF Jobs is the first implementation target, but Modal and RunPod should be able to emit the same structured contracts later.

Target user flow:

1. A cloud stage starts.
2. The stage emits `bootstrap_started`, `artifacts_downloaded`, `model_loaded`, `runtime_ready`, and `work_started` events as they happen.
3. While the stage runs, it emits periodic `progress`, `resource_snapshot`, and `artifacts_synced` events.
4. `cloud-jobs show` and `cloud-jobs logs` can summarize health from the structured event stream first.
5. On failure, the stage emits a single structured `failed` event with normalized error fields, while raw logs remain available for debugging.
6. Agents can inspect one stable schema instead of re-parsing stage-specific noise.

#### Design Approach

Use one canonical stage-event schema with a small set of event types:

- `bootstrap_started`
- `deps_installed`
- `repo_ready`
- `artifacts_downloaded`
- `model_loaded`
- `runtime_ready`
- `work_started`
- `progress`
- `resource_snapshot`
- `artifacts_synced`
- `completed`
- `failed`

Every event will carry a common envelope:

```json
{
  "schema_version": 1,
  "timestamp": "2026-03-22T15:55:43.123456+00:00",
  "stage": "evaluation",
  "provider": "hf_jobs",
  "job_ref": "69c010c871691dc46f163a5e",
  "run_prefix": "runs/hf_jobs/sft/20260322_103451-eafd2a89",
  "event": "runtime_ready",
  "status": "running",
  "message": "vLLM server is ready",
  "details": {
    "model": "unsloth/qwen3-4b-unsloth-bnb-4bit",
    "elapsed_sec": 54.2
  }
}
```

And stage-specific details:

**Evaluation**
- `cases_done`
- `cases_total`
- `passed`
- `warned`
- `failed`
- `backend`
- `tokens_in`
- `tokens_out`

**Loss**
- `examples_done`
- `examples_total`
- `batches_done`
- `rows_written`
- `current_batch_tokens`
- `high_loss_count`

**Training**
- `global_step`
- `max_steps`
- `epoch`
- `loss`
- `learning_rate`
- `checkpoint`

**Resource snapshots**
- `gpu_mem_used_gb`
- `gpu_mem_free_gb`
- `gpu_util_pct` when available
- `cpu_mem_used_gb`

#### Artifact Layout

Each stage output root should contain:

```text
logs/
├── stage_events.jsonl          # canonical append-only structured stream
├── stage_summary.json          # latest reduced summary
└── raw.log                     # raw stdout/stderr when collected
```

Per-worker partial files remain isolated:

```text
workers/<worker-id>/...
shards/<worker-id>/...
```

Only the coordinator writes:
- `logs/stage_events.jsonl`
- `logs/stage_summary.json`

This prevents concurrent overwrite problems while preserving a clean operator stream.

#### Health Model

The logger should compute machine-readable health:

- `healthy`
- `degraded`
- `stalled`
- `failed`

Rules:
- `stalled` if no progress event arrives within a configured window
- `degraded` if retries, repeated sync failures, or low memory headroom appear
- `failed` on terminal structured failure

This lets `cloud-jobs show` surface:

```text
Stage: evaluation
Health: healthy
Elapsed: 13m 11s
Model loaded: yes
Runtime ready: yes
Progress: 17 / 77 cases
Pass/Warn/Fail: 11 / 1 / 5
GPU: 7.8 GB used, 14.6 GB free
Last sync: 26s ago
```

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Schema grows into another noisy dump | Medium | High | Keep event types small and versioned |
| Concurrent workers race on shared summary | Medium | High | Only coordinator writes canonical summary |
| Consumer UX drifts from producer schema | Medium | Medium | Add schema + handler integration tests |
| Too much logging overhead slows jobs | Low | Medium | Rate-limit resource snapshots and summary rewrites |
| Raw logs still needed for root cause | High | Low | Preserve `raw.log` side-channel |

## Definition of Done

The implementation is done when:

1. cloud eval and loss jobs emit canonical structured stage events incrementally
2. `cloud-jobs show` gives an immediate operator summary without parsing raw logs
3. failure states are normalized and easy to classify
4. concurrent worker output cannot corrupt stage summaries
5. raw stdout is preserved but no longer required for routine health checks

At that point, agents should be able to inspect live run health from structured artifacts first and use raw logs only for deep debugging.
