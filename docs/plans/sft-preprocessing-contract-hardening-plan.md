# Implementation Plan: SFT Preprocessing Contract Hardening

> Generated by Codex on 2026-04-02
> Status: PENDING APPROVAL

## Summary

Harden the SFT training pipeline by moving preprocessing ownership into the repo instead of depending on implicit TRL or Unsloth dataset preparation behavior. The immediate trigger is the Gemma 4 launch failure where the trainer reaches collation with raw `{"text": ...}` rows instead of tokenized encodings, but the long-term goal is broader: define one canonical SFT data contract, materialize backend-ready datasets explicitly, and make trainer integrations thin adapters rather than places where dataset semantics are inferred.

This plan does **not** propose building a custom full trainer stack. It proposes owning the boundary between:

1. repo-level training semantics
2. model-family chat formatting
3. backend-specific trainer consumption

Target outcome:

- one canonical repo-owned SFT dataset contract
- one checked-in preprocessing pipeline that can produce tokenized, masked training rows deterministically
- thin adapters for TRL/Unsloth and plain Transformers
- local one-batch validation before expensive cloud launches

### 📋 Preparation Phase
**Effort**: Medium

#### Research Findings
- [x] Current Gemma 4 HF Jobs run loads the model successfully after upgrading to source `transformers`, `unsloth`, and `unsloth_zoo`
- [x] The active failure is no longer model support; it is a dataset contract failure at collation time
- [x] Current traceback: `ValueError: ... includes input_ids, but you provided ['text']`
- [x] Current repo path already uses the Gemma tokenizer / pretrained chat template when available
- [x] Current repo path preformats examples to a `text` column, which is an intermediate representation rather than a final trainer-ready representation
- [x] TRL documentation still describes support for `messages` and `text`, but the active Unsloth-patched training path is not reliably converting that representation into encodings before collation in our stack

#### Problem Statement

The current SFT pipeline is ambiguous about who owns preprocessing:

- repo code owns some semantics: tool-call rendering, chat template application, truncation decisions
- trainer libraries are still expected to own some semantics: tokenization, masking, collation preparation

That mixed ownership is brittle. When the library stack changes, hidden assumptions break and failures appear late in cloud runs.

#### Questions to Resolve
- [ ] What should be the canonical repo-level SFT example schema: conversational only, or conversational plus prompt-completion?
- [ ] Should assistant-only masking be represented explicitly in tokenized labels, or delegated to backend-specific collators?
- [ ] Should tool calls remain rendered-to-text in SFT for now, or should the contract preserve native structured tool-call fields for future backends?
- [ ] Which exact backend adapters are first-class on day one: TRL/Unsloth only, or TRL/Unsloth plus plain Transformers?

#### Preparation Outcomes
- The repo should own semantics and preprocessing.
- Upstream trainers should own optimization, distributed execution, checkpointing, and hardware efficiency.
- The stable long-term seam is the preprocessed dataset contract, not trainer-specific implicit behavior.

#### Design Approach

Adopt a three-layer design:

1. **Canonical Example Contract**
   - Repo-owned semantic representation of an SFT example
   - Independent of trainer library

2. **Preprocessing Pipeline**
   - Deterministically transforms canonical examples into trainer-ready features
   - Responsible for:
     - message normalization
     - tool-call rendering or preservation
     - chat templating
     - tokenization
     - masking
     - truncation
     - validation

3. **Backend Adapter**
   - Passes already prepared data into TRL/Unsloth or Transformers
   - Should not reinvent semantic decisions

#### Core Principle: Own the Contract, Not the Trainer

The repo should not reimplement:

- training loops
- distributed execution
- checkpoint orchestration
- optimizer logic
- mixed precision details

The repo should explicitly own:

- what a training example means
- how tool-call examples are represented
- where loss should be applied
- what tokenization output is expected by a backend

#### Canonical Contract Proposal

Support exactly two primary input shapes:

1. **Conversational SFT**
```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

2. **Prompt-Completion SFT**
```json
{
  "prompt": "... or structured conversational prompt ...",
  "completion": "... or structured assistant response ..."
}
```

Everything else is derived from these two shapes.

#### Materialized Dataset Contract

The preprocessing layer should emit trainer-ready rows such as:

```python
{
    "input_ids": list[int],
    "attention_mask": list[int],
    "labels": list[int],
    "example_format": "messages" | "prompt_completion",
    "loss_mask_mode": "full_sequence" | "assistant_only",
    "truncation_applied": bool,
    "source_hash": str,
}
```

Optional debug-only metadata can be retained in sidecar fields or lineage, but the trainer-facing contract should be explicit and minimal.

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Canonical SFT formats | Messages only vs messages + prompt/completion | Messages + prompt/completion | Covers current conversational data and future prompt-completion workflows without schema sprawl |
| Ownership of tokenization | Repo vs TRL/Unsloth | Repo | Most robust seam; avoids backend magic drift |
| Ownership of masking | Repo vs collator | Repo | Makes assistant-only loss deterministic and testable |
| Tool-call representation | Render to text vs preserve structure | Start with render-to-text in canonical conversational preprocessing, leave room for future structured tool-call variant | Minimizes immediate breakage while retaining extensibility |
| Trainer backend strategy | Custom trainer vs thin adapters | Thin adapters | Lower maintenance, keeps leverage from upstream optimizations |
| Validation point | Cloud only vs local preflight | Local preflight required | Fail fast before expensive A100 jobs |

#### Interface Contracts

**Canonical normalization**
```python
def normalize_sft_example(raw_example: dict) -> CanonicalSFTExample:
    ...
```

**Backend-independent preprocessing**
```python
def materialize_sft_features(
    example: CanonicalSFTExample,
    tokenizer,
    max_seq_length: int,
    loss_mask_mode: str,
    tool_call_mode: str,
) -> dict[str, list[int] | str | bool]:
    ...
```

**Dataset-level transformation**
```python
def prepare_sft_dataset(
    dataset,
    tokenizer,
    max_seq_length: int,
    loss_mask_mode: str,
    backend: str,
) -> Dataset:
    ...
```

**Trainer adapter**
```python
def build_sft_trainer(
    model,
    training_args,
    train_dataset,
    eval_dataset,
    tokenizer,
    backend: str = "trl_unsloth",
):
    ...
```

| File | Changes |
|------|---------|

| `Trainers/sft/train_sft.py` | Consume pretokenized datasets and explicit labels instead of implicit `text`/`messages` trainer prep |
| `shared/experiment_tracking/lineage_enrichment.py` or related lineage builders | Capture preprocessing contract metadata |

#### Implementation Sequence

1. Define `CanonicalSFTExample` and preprocessing invariants
2. Add normalization for current repo data shapes:
   - `conversations` → `messages`
   - `messages` passthrough
   - prompt/completion direct support
3. Add chat-template application using the model tokenizer
4. Add explicit tokenization and truncation
5. Add explicit label construction for:
   - full-sequence loss
   - assistant-only loss
6. Switch `train_sft.py` to consume tokenized rows
7. Keep a thin TRL/Unsloth adapter that no longer depends on automatic dataset preparation
8. Record preprocessing metadata in training lineage
9. Add local preflight checks and fail before training if dataset rows are not trainer-ready

#### Migration Strategy

Phase the change instead of replacing everything at once:

1. **Phase 1: Contract introduction**
   - Add canonical preprocessing functions and tests
   - Keep current code paths available behind flags if needed

2. **Phase 2: SFT trainer migration**
   - Make tokenized datasets the default path
   - Retire implicit `text` handoff for cloud SFT

3. **Phase 3: Cleanup**
   - Remove stale compatibility shims
   - Remove ambiguous “maybe backend will preprocess this” code paths

#### Explicit Non-Goals

- Building a bespoke full training loop
- Replacing TRL or Unsloth entirely
- Solving future native structured tool-calling training in the same change set
- Rewriting KTO or GRPO in this phase

| Risk | Likelihood | Impact | Mitigation |

| New preprocessing path changes behavior for previously stable Qwen runs | Medium | High | Add golden fixtures from known-good runs and compare tokenization/masking outputs |
| Assistant-only masking semantics diverge from prior implicit trainer behavior | Medium | High | Define masking explicitly, add tests against known examples, record mode in lineage |
