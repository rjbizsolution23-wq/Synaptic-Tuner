# Implementation Plan: Cloud Dependency Management

> Generated by `/PACT:plan-mode` on 2026-03-14
> Status: IN_PROGRESS

## Summary

Fix cloud training dependency management across all three providers (HF Jobs, Modal, RunPod) by adopting a **hybrid strategy**: use unsloth's official Docker image for HF Jobs and RunPod (eliminating both the torch version conflict and missing git issue), while keeping Modal's declarative pip approach with properly pinned versions. A shared configuration section in `cloud_config.yaml` provides a single source of truth for image tags and version pins.

#### Design Approach

**Hybrid strategy** — two complementary approaches united by shared config:

1. **HF Jobs + RunPod**: Use `unsloth/unsloth` Docker image directly
   - Eliminates transitive dependency resolution (unsloth, torch, CUDA all pre-installed)
   - Only install minimal project-specific deps on top: `pyyaml wandb hf_transfer python-dotenv`
   - Git is pre-installed — no `apt-get` needed
   - Command chain simplifies to: `pip install <project-deps> && git clone ... && cd repo && python train.py`

2. **Modal**: Keep declarative `Image.pip_install()` with properly pinned versions
   - Modal's image caching makes this efficient (layers cached between runs)
   - Already has `.apt_install("git")`
   - Pin to exact versions following Modal's official unsloth example pattern

3. **Shared config**: `cloud_config.yaml` gains a `dependencies` section specifying:
   - Docker image tag (for HF Jobs/RunPod)
   - Pinned pip versions (for Modal and as documentation)
   - Project-specific deps list (shared across all providers)

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Image strategy | A) Unsloth Docker image, B) Generic pytorch + pinned pip, C) Custom Dockerfile | A for HF Jobs/RunPod, pinned pip for Modal | Eliminates both bugs; unsloth team maintains compatibility; user suggested this |
| Image tag | A) `latest`/`stable`, B) Pinned version tag | B: `2026.1.2-pt2.9.0-cu12.8-update` | Immutable, reproducible, no surprise upgrades |
| Where deps live | A) Code constant, B) requirements.txt, C) cloud_config.yaml | C: cloud_config.yaml + minimal code constant for project deps | Single source of truth in config; code reads from it |
| RunPod image | Keep CUDA 11.8 vs upgrade | Upgrade — current image uses unsloth Docker image (CUDA 12.8) | Current `runpod/pytorch:2.1.0-py3.10-cuda11.8.0` is 3 versions behind; likely cause of user's torch conflict |
| `setup_commands` in YAML | Keep vs remove vs replace | Replace with `extra_setup_commands` (empty by default) | Standard deps come from image; escape hatch for user customization |

#### Interface Contracts
- `cloud_config.yaml` → `load_cloud_config()` → `CloudTrainingConfig` — adds `image` and `dependencies` fields
- Each backend reads image from config, falls back to provider-specific default
- Project-specific pip deps defined once in config, consumed by all backends
