# Implementation Plan: Provider-Agnostic Experiment Orchestration

> Generated by Codex on 2026-03-21
> Status: PENDING APPROVAL

## Summary

Create one provider-agnostic experiment system that starts from a single experiment config, launches training on the selected provider, captures run telemetry while training, then automatically performs post-run evaluation and per-example loss inference, and finally stores everything in a canonical experiment record plus ML-ready analysis artifacts and agent-readable analysis bundles.

The system should treat Hugging Face Jobs as the first provider implementation, not the core abstraction. Modal and RunPod must fit into the same contracts later without schema changes.

Target user flow:

1. User defines experiment config settings.
2. System launches the run on the selected provider.
3. System captures training metadata and live telemetry while the run is active.
4. On successful training completion, system launches evaluation and per-example loss inference.
5. System organizes all resulting data so LightGBM or another tabular learner can consume it directly.
6. An agent can analyze the experiment outputs through stable scripts and skill-guided summaries, form a hypothesis, and launch a follow-up run without reverse-engineering raw artifacts.

This is a follow-on consolidation plan. It builds on the implemented unified run registry and the planned SFT experiment pipeline, but makes the **experiment** itself the top-level product instead of stitching together multiple side systems.

#### Design Approach

The architecture should separate concerns cleanly:

1. **ExperimentSpec** — declarative user intent
2. **ExperimentOrchestrator** — lifecycle engine
3. **ProviderAdapter** — launch/poll/fetch semantics per provider
4. **TrackingService** — canonical writes for runs, experiments, and links
5. **FeatureBuilder** — post-run tabular dataset assembly
6. **AnalysisBundleBuilder** — stable, agent-readable views over experiment state and results

#### Core Principle: Unified System, Not Unified Storage Backend

The system is unified if all providers write through the same orchestration and tracking contracts. It does **not** require all artifacts to live in one filesystem.

Canonical ownership model:

- provider-native artifact root is the source of truth for raw artifacts
- `registry.jsonl` is the source of truth for runs
- `experiment.json` is the source of truth for experiment grouping and derived outputs

#### Architecture Diagram

```text
User config (ExperimentSpec YAML / CLI)
        │
        ▼
ExperimentOrchestrator
        │
        ├── create/attach Experiment
        ├── launch training via ProviderAdapter
        ├── capture live telemetry
        ├── register training run
        ├── launch evaluation
        ├── launch per-example loss inference
        ├── materialize feature dataset
        └── mark experiment complete
        │
        ├──────────────► TrackingService
        │                 ├── RunRegistry (.tracking/registry.jsonl)
        │                 ├── Experiment record (.tracking/experiments/<id>/experiment.json)
        │                 └── run links / statuses / derived outputs
        │
        └──────────────► ProviderAdapter
                          ├── HF Jobs
                          ├── Modal
                          └── RunPod
```

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Top-level abstraction | A) Run B) Experiment C) Provider | **B) Experiment** | User intent is “run an experiment,” not “append a run record” |
| Provider integration | A) HF-first schema B) Provider-neutral schema | **B) Provider-neutral schema** | Prevents HF lock-in; Modal/RunPod should slot in without schema churn |
| Artifact addressing | A) Local paths only B) Provider-native roots | **B) Provider-native roots** | Cloud should stay cloud; local materialization remains optional |
| Telemetry storage | A) Logs only B) Structured metrics timeline | **B) Structured metrics timeline** | Required for analysis and later capacity models |
| Loss computation | A) Optional side tool B) Standard experiment stage | **B) Standard experiment stage** | Needed for quality modeling and LightGBM workflows |
| Feature dataset | A) Per-run B) Per-experiment | **B) Per-experiment** | Cross-run comparison is the main use case for tabular learning |
| Experiment creation | A) Manual only B) Auto-create with override | **B) Auto-create with override** | Default should be one command for the full lifecycle |
| Agent analysis interface | A) Raw artifacts only B) Canonical analysis bundle + scripts | **B) Canonical analysis bundle + scripts** | Agents should not need to infer structure from ad hoc logs every time |

#### Interface Contracts

**1. ExperimentSpec**
```yaml
experiment:
  name: "battle-smoke-smollm2"
  objective: "train_eval_loss_smoke"
  provider: "hf_jobs"         # later: modal, runpod
  method: "sft"               # later: kto, grpo
  dataset:
    source: "professorsynapse/claudesidian-synthetic-dataset"
    file: "nonthinking_tools_sft_03.14.26_with_toolmanager_loop_no_gettools_privacy_v1.3.jsonl"
  training:
    model_name: "HuggingFaceTB/SmolLM2-1.7B-Instruct"
    max_steps: 20
    gpu: "a10g-small"
  evaluation:
    preset: "quick"
  loss:
    enabled: true
  features:
    enabled: true
    format: ["jsonl", "csv", "parquet"]
```

**2. Provider-neutral RunRecord additions**
```python
provider: str | None
artifact_backend: str | None
artifact_root: str | None      # local path or provider-native URI
job_ref: str | None            # provider-native execution identifier
source_commit: str | None
experiment_id: str | None
stage: str | None              # training | evaluation | loss | compare | judge
```

**3. Canonical Experiment record**
```python
experiment_id: str
name: str
created_at: str
provider: str
method: str
objective: str
dataset_source: str
base_model_name: str
run_ids: list[str]
training_run_id: str | None
evaluation_run_id: str | None
loss_run_id: str | None
selected_run_id: str | None
status: str
artifact_roots: dict[str, str]
derived_outputs: dict[str, str]   # features.csv, features.parquet, judge_scores.jsonl, etc.
```

**4. Feature dataset contract**

The experiment output must include a canonical ML-ready table with:

- experiment metadata
- training config features
- provider/hardware/runtime features
- live telemetry aggregates
- final training/eval outcomes
- per-example loss features
- optional judge labels

Formats:
- `features.jsonl`
- `features.csv`
- `features.parquet`

The schema must be algorithm-agnostic. LightGBM is the first consumer, not the only one.

**5. Canonical analysis bundle contract**

Each completed experiment should also emit an agent-readable bundle such as:

```text
.tracking/experiments/<id>/analysis/
├── experiment_summary.json
├── experiment_summary.md
├── run_matrix.csv
├── feature_dataset.parquet
├── hypothesis_context.json
├── next_run_candidates.json
└── failure_slices/
    ├── eval_failures.jsonl
    └── high_loss_examples.jsonl
```

Purpose:
- `experiment_summary.json` gives a compact machine-readable overview
- `experiment_summary.md` is a human and agent scanning surface
- `run_matrix.csv` compares runs side by side
- `hypothesis_context.json` captures the structured facts needed for “why did this run behave this way?”
- `next_run_candidates.json` stores ranked suggested config changes for the next experiment
- `failure_slices/*.jsonl` gives concrete examples to inspect without loading the full corpus

| Concern | Status | Notes |
