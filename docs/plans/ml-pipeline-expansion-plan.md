## Summary

Expand Toolset-Training from a pure LLM fine-tuning platform into a dual-purpose model training platform supporting both LLM fine-tuning (SFT/KTO/GRPO) and traditional ML (LightGBM, XGBoost, scikit-learn, AutoML). The implementation follows a 6-phase roadmap across three tracks: Foundation infrastructure, Pipeline-enhancing ML (quality scoring, training data influence), and Standalone ML training for arbitrary prediction tasks.

#### Design Approach

**Architecture Principles**:
1. **Additive, not invasive** вЂ” New modules, minimal changes to existing code
2. **Config-driven** вЂ” YAML in, trained model out (matches existing philosophy)
3. **Strategy pattern** for algorithms вЂ” Registry + ABC ensures uniform interface
4. **Pipeline pattern** for features вЂ” sklearn Pipeline/ColumnTransformer from config
5. **Facade pattern** for tracking вЂ” ExperimentTracker abstracts MLflow

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Config validation | A) Dataclasses (existing pattern) B) Pydantic C) Plain dict | **B) Pydantic** | User-facing YAML needs rich validation with clear error messages; Pydantic v2 is fast and gives schema validation for free |
| Algorithm registry | A) Protocol B) ABC C) Simple dict | **B) ABC + dict registry** | ABC enforces interface at instantiation; dict provides name-based lookup from config (Strategy pattern) |
| Module location | A) `Trainers/ml/` B) `Trainers/ml_classifiers/` | **A) `Trainers/ml/`** | More general вЂ” covers classification, regression, clustering. Shorter paths |
| Shared code split | A) All in `shared/ml/` B) All in `Trainers/ml/` C) Split | **C) Split** | If SynthChat/Evaluator need it в†’ `shared/ml/`. If only Trainers/ml/ needs it в†’ `Trainers/ml/` |
| MLflow dependency | A) Required B) Optional C) Optional with JSON fallback | **C) Optional with fallback** | `metrics.json` always written (zero-dep). MLflow opt-in via config. Users can train without MLflow installed |
| Feature engineering | A) Config-only B) Config + custom transformers C) Plugin system | **B) Config + custom** | YAML handles 90% of cases. Custom sklearn transformers for project-specific features (e.g., ConversationFeatureExtractor) |
| Output directory | A) Project root B) Inside `Trainers/ml/` | **B) `Trainers/ml/ml_output/`** | Mirrors SFT pattern; configurable via `output.dir` in YAML |
| Pipeline ML code location | A) In integration points B) In `Trainers/ml/` C) Hybrid | **C) Hybrid** | Training code in `Trainers/ml/pipeline_tasks/`. Integration hooks (10-line glue) in SynthChat/shared/llm/ |
| AutoML engine | A) FLAML B) AutoGluon C) PyCaret | **A) FLAML** (primary), B optional | Lightweight, budget-aware, extensible. AutoGluon optional for max accuracy. PyCaret too opinionated |

#### Interface Contracts (Define Before Code)

TuningConfig: enabled, method, n_trials, timeout, search_space

EvalConfig: metrics, generate_plots, shap_analysis
OutputConfig: dir, save_model, save_pipeline

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Framework | pytest + parametrize + markers | De facto standard; parametrize critical for multi-algorithm testing |
| ML quality testing | Smoke tests in CI, threshold tests for nightly | Avoids flaky tests while ensuring models train correctly |
| MLflow in tests | Mocked (unit), file backend (integration) | No server dependency |
| GPU tests | `@pytest.mark.gpu` + auto-skip | CI without GPU, local GPU validation |
| Fixture size | 50 rows (committed), generated for integration | Fast, reproducible, debuggable |
| Determinism | Fixed seeds everywhere + `np.allclose()` | Handles floating-point variance |
