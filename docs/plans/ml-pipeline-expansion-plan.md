# Implementation Plan: ML Pipeline Expansion

> Generated by `/PACT:plan-mode` on 2026-03-12
> Status: IN_PROGRESS
> User Decisions: Same conda env; Phase 0→1→2 priority (standalone ML first)

## Summary

Expand Toolset-Training from a pure LLM fine-tuning platform into a dual-purpose model training platform supporting both LLM fine-tuning (SFT/KTO/GRPO) and traditional ML (LightGBM, XGBoost, scikit-learn, AutoML). The implementation follows a 6-phase roadmap across three tracks: Foundation infrastructure, Pipeline-enhancing ML (quality scoring, training data influence), and Standalone ML training for arbitrary prediction tasks.

#### Design Approach

**Architecture Principles**:
1. **Additive, not invasive** — New modules, minimal changes to existing code
2. **Config-driven** — YAML in, trained model out (matches existing philosophy)
3. **Strategy pattern** for algorithms — Registry + ABC ensures uniform interface
4. **Pipeline pattern** for features — sklearn Pipeline/ColumnTransformer from config
5. **Facade pattern** for tracking — ExperimentTracker abstracts MLflow

**Component Diagram**:
```
tuner.py (CLI)
  │
  ├── Trainers/ml/train.py (orchestrator)
  │     ├── configs/ (YAML, validated by Pydantic)
  │     ├── algorithms/registry.py (Strategy pattern)
  │     │     ├── lightgbm_wrapper.py
  │     │     ├── xgboost_wrapper.py
  │     │     ├── auto_wrapper.py (FLAML)
  │     │     └── ... (one per algorithm)
  │     ├── features/builder.py (Config → ColumnTransformer)
  │     ├── data/loader.py (multi-format ingestion)
  │     ├── evaluation/ (metrics, plots, SHAP, reports)
  │     └── ml_output/ (timestamped run dirs)
  │
  ├── shared/ml/ (reusable utilities)
  │     ├── features.py (conversation feature extraction)
  │     ├── data_loaders.py (JSONL → tabular)
  │     └── metrics.py (shared metric helpers)
  │
  ├── shared/experiment_tracking/ (MLflow facade)
```

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Config validation | A) Dataclasses (existing pattern) B) Pydantic C) Plain dict | **B) Pydantic** | User-facing YAML needs rich validation with clear error messages; Pydantic v2 is fast and gives schema validation for free |
| Algorithm registry | A) Protocol B) ABC C) Simple dict | **B) ABC + dict registry** | ABC enforces interface at instantiation; dict provides name-based lookup from config (Strategy pattern) |
| Module location | A) `Trainers/ml/` B) `Trainers/ml_classifiers/` | **A) `Trainers/ml/`** | More general — covers classification, regression, clustering. Shorter paths |
| Shared code split | A) All in `shared/ml/` B) All in `Trainers/ml/` C) Split | **C) Split** | If SynthChat/Evaluator need it → `shared/ml/`. If only Trainers/ml/ needs it → `Trainers/ml/` |
| MLflow dependency | A) Required B) Optional C) Optional with JSON fallback | **C) Optional with fallback** | `metrics.json` always written (zero-dep). MLflow opt-in via config. Users can train without MLflow installed |
| Feature engineering | A) Config-only B) Config + custom transformers C) Plugin system | **B) Config + custom** | YAML handles 90% of cases. Custom sklearn transformers for project-specific features (e.g., ConversationFeatureExtractor) |
| Output directory | A) Project root B) Inside `Trainers/ml/` | **B) `Trainers/ml/ml_output/`** | Mirrors SFT pattern; configurable via `output.dir` in YAML |
| Pipeline ML code location | A) In integration points B) In `Trainers/ml/` C) Hybrid | **C) Hybrid** | Training code in `Trainers/ml/pipeline_tasks/`. Integration hooks (10-line glue) in SynthChat/shared/llm/ |
| AutoML engine | A) FLAML B) AutoGluon C) PyCaret | **A) FLAML** (primary), B optional | Lightweight, budget-aware, extensible. AutoGluon optional for max accuracy. PyCaret too opinionated |

#### Interface Contracts (Define Before Code)

**1. TrainingConfig (Pydantic)**
```
TaskConfig: type, name, target_column, eval_metric, random_state
DataConfig: train_path, test_path/test_size, stratify
FeaturesConfig: numeric, categorical, text, drop_columns
AlgorithmConfig: name, params (plain dict)
TuningConfig: enabled, method, n_trials, timeout, search_space
CVConfig: method, n_splits, n_repeats
EvalConfig: metrics, generate_plots, shap_analysis
OutputConfig: dir, save_model, save_pipeline
TrackingConfig: enabled, backend, experiment_name, tags
```

**2. AlgorithmWrapper (ABC)**
```
create_estimator(task_type, params) → sklearn.BaseEstimator
get_search_space(task_type) → dict
name: str
supports_classification: bool
supports_regression: bool
supports_gpu: bool
```

**3. ExperimentTracker (ABC)**
```
init(config) → None
start_run(run_name) → context_manager
log_params(params) → None
log_metrics(metrics) → None
log_model(model, path) → None
log_artifact(path) → None
```

**4. build_preprocessor(FeaturesConfig) → ColumnTransformer**

**5. Output Directory Structure**
```
ml_output/YYYYMMDD_HHMMSS/
├── config.yaml          # Copy of training config
├── model.joblib         # Serialized pipeline (preprocessing + model)
├── metrics.json         # All evaluation metrics
├── schema.json          # Feature schema (column names, types)
├── plots/               # Confusion matrix, ROC, feature importance, SHAP
│   ├── confusion_matrix.png
│   ├── roc_curve.png
│   ├── feature_importance.png
│   └── shap_summary.png
└── logs/
    └── training.log     # Training log
```

#### Key Testing Decisions
| Decision | Choice | Rationale |
|----------|--------|-----------|
| Framework | pytest + parametrize + markers | De facto standard; parametrize critical for multi-algorithm testing |
| ML quality testing | Smoke tests in CI, threshold tests for nightly | Avoids flaky tests while ensuring models train correctly |
| MLflow in tests | Mocked (unit), file backend (integration) | No server dependency |
| GPU tests | `@pytest.mark.gpu` + auto-skip | CI without GPU, local GPU validation |
| Fixture size | 50 rows (committed), generated for integration | Fast, reproducible, debuggable |
| Determinism | Fixed seeds everywhere + `np.allclose()` | Handles floating-point variance |

```
Phase 4: CLI + Pipeline Integration ──────────────────────────────────
  ├── tuner.py ML subcommand (train, eval, list, mlflow)
  ├── run.sh ml commands
  ├── Quality scorer training (LightGBM + TF-IDF on JSONL)

  ├── PerExampleLossCallback for SFT/KTO trainers
```

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Scope creep into framework territory | High | High | Strict phase boundaries — each phase delivers standalone value. Don't start N+1 until N is usable |
| Config complexity overwhelms users | Medium | Medium | Ship minimal templates; support `--auto` mode needing only data path + target column |
| Dependency conflicts with conda env | Low | High | Test in isolated env first; all libs well-tested with PyTorch |
| No existing test culture | Medium | Medium | Keep tests simple; provide templates; add `./run.sh test` |
| ML non-determinism causes flaky tests | Medium | Low | Fixed seeds everywhere; `np.allclose()` for assertions; separate deterministic from stochastic tests |
| Training data influence (TracIn) correctness with LoRA | Medium | Medium | Start with loss trajectory only (trivial); TracIn is Phase 5 work requiring careful validation |
| Feature engineering quality for quality scorer | Medium | Medium | Start with known-good features from validation module; iterate with SHAP |
| Two paradigms confuse users | Medium | Medium | Clear CLI separation; numbered menu options; good docs |
