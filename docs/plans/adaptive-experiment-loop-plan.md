## Summary

Build the next version of the experiment loop so one experiment spec can:

1. train on the selected provider
2. run evaluation and per-example loss robustly after training
3. use hardware-adaptive batching/concurrency instead of hardcoded worker counts
4. save all outputs in a stable experiment bundle
5. derive concrete next-run recommendations from eval + loss + telemetry data
6. optionally emit a draft next experiment spec that an agent can review and launch

This plan is intentionally broader than pure “make eval faster.” The next run needs three things at once:

- reliable stage recovery
- speed via adaptive parallelism and batching
- analysis outputs strong enough to support the next experiment hypothesis automatically
- explicit stage controls so operators can run only the pieces they want when needed

This builds on the already landed experiment tracking, analysis bundles, eval-with-loss wiring, and stale-stage recovery. It does **not** assume Hugging Face Jobs is the only provider. The control layer should be provider-agnostic even though HF is the first implementation target.

Target user flow:

1. User runs `run-experiment` with a single experiment spec.
2. Training completes.
3. By default, the system continues through the full post-training pipeline: eval, exact loss, analysis, and recommendation.
4. Post-training runtime chooses an eval/loss execution strategy based on available hardware and runtime backend.
5. Every data-producing stage writes artifacts incrementally as batches complete, so partial results survive crashes.
6. The experiment bundle writes structured failure slices, feature tables, telemetry summaries, and recommendation outputs.
7. An agent loads the finished experiment, reviews the recommendation set, and launches the next candidate run with minimal manual interpretation.

#### Questions Resolved
- [x] Should eval and loss stay separate jobs forever? No. They should be one post-training execution family, with embedded loss when possible.
- [x] Should parallelism be statically configured? No. Defaults should be adaptive to runtime capacity.
- [x] Is there an existing control strategy for adaptive concurrency? Yes. AIMD-style concurrency control plus warmup probing and hysteresis is the pragmatic choice here.
- [x] Should the next-run planner depend only on aggregate scores? No. It must incorporate failure slices, tag-level breakdowns, loss distribution, and training capacity telemetry.
- [x] Should the normal flow gate merge/loss on eval score? No. Full data collection should be the default; stage skipping should be an explicit operator override.
- [x] Should post-training loss keep using Unsloth? No. The target architecture is `vLLM` for eval generation and exact batched `transformers` loss scoring for the dataset pass.
- [x] Should stages buffer all outputs until the end? No. All data-producing stages should flush incrementally and be resumable from partial artifacts.

#### Preparation Outcomes
- The current pipeline can complete train → eval → loss, but the runtime is still mostly serial and the next-run recommendations are intentionally weak.
- Eval is still case-serial in `Evaluator/runner.py`.
- Loss is still example-serial in `per_example_loss.py`.
- The experiment bundle already has a usable home for hypothesis and recommendation outputs, but the logic behind those outputs is too shallow for the next run loop.

#### Design Approach

The loop should be split into five layers:

1. **Experiment orchestration**
   - knows which stages exist
   - tracks status, recovery, artifact roots, and next actions

2. **Runtime autotuning**
   - measures memory and throughput
   - decides concurrency, batch size, and token budget
   - adjusts during the run

3. **Execution engines**
   - eval engine for prompt/case workloads
   - exact loss engine for dataset-wide forward passes

4. **Recommendation engine**
   - reads completed experiment bundle
   - ranks hypotheses and next-run candidates
   - writes machine-readable and human-readable outputs

5. **Stage selection**
   - full pipeline is the default
   - operators can still run only selected stages for smoke tests, debugging, or recovery

6. **Crash-safe persistence**
   - every stage writes partial artifacts as batches complete
   - concurrent workers write to isolated shards, not a shared hot file
   - final outputs are assembled by deterministic merge/promote steps

#### Core Principle: Adaptive By Default, Override When Needed

The system should not require manual worker-count tuning for common runs.

Default behavior:
- inspect runtime capacity
- run a short warmup
- select initial concurrency/batch settings
- adapt while the job runs

Manual overrides should still exist for debugging or benchmarking, but they should be optional.

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Eval acceleration algorithm | A) fixed workers B) warmup + AIMD C) RL/bandit | **B) warmup + AIMD** | Simple, stable, widely used for adaptive concurrency |
| Loss acceleration algorithm | A) one-example-at-a-time B) fixed batch size C) adaptive token-batch control | **C) adaptive token-batch control** | Works better across model sizes and sequence-length variance |
| Runtime signal source | A) config only B) VRAM only C) VRAM + latency + tokens/sec + OOM signals | **C)** | Prevents overfitting to one metric |
| Eval engine | A) Unsloth only B) vLLM only C) backend-selectable | **C)** | Keep a stable fallback while making vLLM the preferred fast eval path |
| Exact loss engine | A) Unsloth B) vLLM C) `transformers` batched forward pass | **C)** | Best fit for exact masked cross-entropy over the full dataset |
| Full-pipeline policy | A) gated by eval score B) full by default with overrides C) manual-only composition | **B)** | Preserves negative signal for analysis and LightGBM while keeping operator control |
| Merge policy | A) always merge first B) never merge C) configurable / auto | **C)** | Keeps smoke runs light and allows a consistent merged inference artifact when useful |
| Partial-write strategy | A) end-of-run only B) shared append file C) shard-per-worker + atomic promote | **C)** | Safest under concurrency and crashes |
| Recommendation source | A) aggregate score only B) heuristics only C) heuristics + structured features + optional ML scorer | **C)** | Good now, extensible later |
| Next-run artifact | A) prose only B) JSON only C) JSON + YAML spec draft + markdown summary | **C)** | Agents and humans both need a stable surface |

#### Adaptive Control Strategy

Use a pragmatic two-phase controller:

**Phase A: Warmup calibration**
- probe available VRAM
- run a few small eval requests or a few small forward batches
- estimate:
  - tokens/sec
  - latency per request
  - memory reserved / free
  - safe token budget

**Phase B: Runtime adaptation**
- use additive increase when:
  - p95 latency remains under threshold
  - VRAM headroom remains above floor
  - no OOM / allocator retry / throttling signals appear
- use multiplicative decrease when:
  - OOM occurs
  - latency spikes beyond guardrail
  - free memory drops below threshold
  - server begins rejecting or stalling requests

This is essentially AIMD with memory guardrails and hysteresis.

For loss:
- adapt on **token budget per batch**, not raw row count
- pack examples into a token-budgeted microbatch
- adjust target token budget up/down based on headroom and step time

#### Interface Contracts

**1. Experiment spec additions**
```yaml
execution:
  stages: ["training", "evaluation", "loss", "analysis", "recommendation"]
  from_stage: null
  only_stage: null
  skip_stages: []

evaluation:
  enabled: true
  runtime: "vllm"
  adaptive_parallelism: true
  target_latency_ms: 4000
  min_workers: 1
  max_workers: 16
  max_tokens_per_request: 1024

loss:
  enabled: true
  runtime: "transformers_exact"
  adaptive_batching: true
  min_batch_tokens: 4096
  max_batch_tokens: 65536
  target_vram_headroom_gb: 2.0

post_training:
  merge_for_inference: "auto"   # auto | true | false
```

**2. Runtime telemetry contract**
```json
{
  "eval_runtime": "vllm",
  "loss_runtime": "transformers_exact",
  "gpu_name": "NVIDIA A10G",
  "gpu_count": 1,
  "vram_total_gb": 22.0,
  "vram_free_gb": 7.3,
  "concurrency": 4,
  "batch_tokens": 24576,
  "tokens_per_second": 812.4,
  "p95_latency_ms": 3210,
  "oom_events": 0
}
```

**3. Incremental-write contract**
```text
training/
├── logs/training_latest.jsonl                 # append-only stream
├── metrics/steps/part-00000.jsonl            # optional sharded metrics
└── manifests/training_state.json             # checkpoint/progress manifest

evaluation/
├── logs/eval_progress.jsonl                  # append-only ordered progress log
├── records/shard-00000.jsonl                 # one shard per worker/batch writer
├── partial/evaluation_results.partial.json   # periodically promoted aggregate snapshot
└── manifests/eval_state.json                 # shard registry + watermark

analysis/loss/
├── shards/losses-00000.jsonl                 # one shard per worker/batch writer
├── partial/loss_summary.partial.json         # periodically updated aggregate summary
├── partial/high_loss_examples.partial.jsonl  # bounded rolling slice
└── manifests/loss_state.json                 # shard registry + completion watermark
```

Rules:
- never have concurrent workers append to the same hot JSONL file
- workers write isolated shard files or batch files
- aggregators merge shard state into partial summaries on an interval
- final artifacts are produced by atomic rename/promote after successful merge
- if a job crashes, the manifest and shards are sufficient for resume/recovery

**4. Stage-selection CLI contract**
```bash
python3 tuner.py run-experiment --experiment-spec ...                    # full pipeline
python3 tuner.py run-experiment --experiment-spec ... --only-stage eval
python3 tuner.py run-experiment --experiment-spec ... --only-stage loss
python3 tuner.py run-experiment --experiment-spec ... --skip-stage loss
python3 tuner.py run-experiment --experiment-spec ... --from-stage eval
```

**5. Recommendation bundle contract**
```text
.tracking/experiments/<id>/analysis/
├── experiment_summary.json
├── experiment_summary.md
├── run_matrix.csv
├── feature_dataset.jsonl

```

**6. Next-run recommendation schema**

### Implementation Sequence

1. **Stabilize post-training execution**
- keep the already landed retry/recovery fixes
- ensure completed eval + retried loss is a first-class supported path
- write explicit runtime metadata for eval and loss stages
- make full-pipeline execution the default contract
- add explicit `only-stage`, `from-stage`, and `skip-stage` style controls
- define common incremental-write and resume manifests for every stage

2. **Batch the loss path**
- change `per_example_loss.py` from one-example forward passes to exact token-budgeted microbatches
- move post-training loss loading to plain `transformers` instead of Unsloth
- add optional cached tokenized datasets to avoid redoing chat templating on repeated passes
- flush partial loss outputs incrementally per batch/shard
- track throughput and memory metrics

3. **Parallelize eval on stable runtime**
- make `Evaluator/runner.py` honor `parallel` and `max_workers`
- preserve ordered result writing and crash-safe progress logs
- gate concurrency by backend capability
- write per-worker shards and merge snapshots instead of shared concurrent appends

4. **Add adaptive runtime controller**
- implement warmup probes
- implement AIMD-based concurrency control
- implement batch-token controller for loss
- write telemetry to analysis bundle

5. **Add dedicated vLLM cloud path**
- finish `cloud_hf_job_vllm.py`
- start vLLM server in a dedicated image
- use concurrent requests for prompt-case evaluation
- keep exact `transformers` loss after eval
- support merged inference artifacts when configured, but do not require merge for every run

6. **Strengthen recommendation engine**
- move beyond “collect another run”
- derive next-run proposals from:
  - tag-level failures
  - failure slices
  - high-loss examples
  - training capacity/oom telemetry
  - prior run deltas

7. **Draft next-run spec automatically**
- emit `draft_next_experiment.yaml`
- wire CLI to:
  - analyze completed experiment
  - print ranked candidates
  - optionally launch the top candidate

## ⚠️ Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Adaptive concurrency causes unstable eval results | High | Keep deterministic result assembly and backend-safe concurrency gating |
| Aggressive batching causes OOMs during loss | High | Warmup probe + multiplicative backoff + partial flush checkpoints |
| vLLM path diverges from stable artifact contract | Medium | Keep output schema identical across eval backends |
| Concurrent writers overwrite or corrupt artifacts | High | Per-worker shard files, manifest-driven merges, atomic promote/rename |
| Recommendation engine overfits to one run | High | Require confidence thresholds and cross-run context before high-confidence recommendations |
| “Adaptive” defaults hide important behavior | Medium | Log controller decisions and persist them in analysis outputs |

## ✅ Definition of Done

This plan is done when:

1. A single experiment spec can complete train → eval → loss without manual stage surgery.
2. Loss computation runs as exact batched `transformers` scoring and records runtime telemetry.
3. Eval supports real parallel execution or a dedicated `vLLM` fast runtime with artifact parity.
4. The finished experiment bundle includes rich eval/loss/telemetry features.
5. `next_run_candidates.json` contains concrete, structured proposals.
6. `draft_next_experiment.yaml` can be launched directly for the follow-up run.
7. Every data-producing stage writes incremental, crash-safe artifacts rather than only flushing at the end.
8. Operators can selectively run `training`, `evaluation`, `loss`, `analysis`, or `recommendation` without editing the spec by hand.
9. An agent can inspect the completed experiment and launch the next run without reverse-engineering raw logs or bucket trees.
