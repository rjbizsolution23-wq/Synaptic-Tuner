# Implementation Plan: Docker-First Local Runtime

> Generated by Codex on 2026-04-09
> Status: PENDING APPROVAL

## Summary

Make local Docker the preferred runtime for GPU-backed work in this repo by adding a first-class local container path for training and evaluation. The goal is not to replace every host-based workflow, but to move the fragile dependency stack out of the Windows/conda environment and into pinned containers where the repo can control `torch`, `transformers`, `trl`, `peft`, `unsloth`, and `vllm` much more predictably.

The proposed local shape is:

- training runs in an official `unsloth/unsloth` container
- fast serving / eval runs in a separate `vllm/vllm-openai` container
- host Python remains for orchestration, editing, docs, and lightweight analysis

This plan also covers better logging for long image pulls so users can tell the difference between a large download and a hung Docker Desktop session.

#### Design Approach

Use Docker as the execution boundary for GPU-dependent code:

1. **Training container**
- official Unsloth image
- pinned tag or digest for reproducibility
- repo mounted as a volume
- outputs written back to the host workspace

2. **Serving container**
- separate `vllm/vllm-openai` image
- used for local eval / OpenAI-compatible serving
- keeps serving dependencies isolated from training dependencies

3. **Host orchestration**
- CLI decides which image to launch
- host handles config parsing, artifact paths, and log streaming
- host Python stays for non-GPU tasks

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Local runtime | A) conda only B) Docker only C) hybrid | **C) hybrid** | Docker becomes the default; conda remains fallback |
| Training image | A) `latest` B) pinned tag C) digest | **C) digest or pinned tag** | Reproducible, avoids surprise upgrades |
| Serving image | A) reuse Unsloth image B) vLLM image | **B) vLLM image** | Cleaner dependency isolation |
| Pull logging | A) plain `docker pull` B) scripted watcher | **B) scripted watcher** | Better signal for slow vs stuck pulls |
| CLI UX | A) hidden helper B) explicit local subcommands | **B) explicit local subcommands** | Makes Docker-first behavior discoverable |

#### Interface Contracts
- `local train` should launch a container with the repo mounted and a writable output directory.
- `local eval` should choose `unsloth` or `vllm` runtime explicitly.
- `local serve` should start a long-lived vLLM container with a clear stop/restart story.
- Pull progress should be surfaced in a way that is visible in PowerShell, not buried behind silent background execution.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Docker Desktop is unavailable or stopped | Medium | High | Detect early and fail with a clear message |
| Large image pull feels stuck | High | Medium | Show visible layer progress and add a periodic status line |
| vLLM and Unsloth need separate images | High | Medium | Treat them as separate local runtimes, not one container |
| Triton mismatch inside Unsloth image | Medium | Medium | Track it with a follow-up smoke test before making it the default |
| Windows bind-mount performance surprises users | Medium | Medium | Document the tradeoff and recommend workspace placement carefully |

## Scope

This plan intentionally starts with local Docker viability and CLI/skill guidance. It does not try to redesign the cloud backend. The cloud image/profile work already exists; the local work should mirror that shape, not invent a separate dependency model.
