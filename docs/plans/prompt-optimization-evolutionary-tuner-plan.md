# Implementation Plan: Prompt Optimization Evolutionary Tuner

> Generated by `pact-codex-plan-mode` on 2026-05-29

Implementation status: schema v2 evolutionary prompt optimization is now
implemented behind `PromptOptimizationService.from_config(...).run()`, exposed
through the standalone tuner command, and consumable by the SynthChat
pre-generation hook via overlays. This document is retained as design history;
the sections below describe the implemented direction, not pending approval.

Current workflow decisions: expose prompt optimization both standalone
(`python tuner.py prompt-optimize --prompt-opt-config ...`) and as an optional
SynthChat pre-generation step (`python -m SynthChat.run generate
--prompt-opt-config ...`). Keep the checked-in lab-kit epistemic humility
Evaluator smoke dry-run safe with `evaluation.evaluator.dry_run: true`; use a
local override or temporary manual copy for real LM Studio smoke. The current
Evaluator adapter injects candidate prompts with `system_overlay`, scores the
configured objective metric (`stats.pass_rate` for the lab-kit smoke), applies
`score_floor` on candidate-scoring failures unless configured to raise, and
emits v2 manifest, candidate history, generation history, lineage, replay, and
SynthChat-compatible overlays. The practical evaluator target threshold is
`stopping.target_score: 0.8`, and the minimal checked-in smoke size is
`population_size: 3`, `max_generations: 1`.

## Summary

Porting `synaptic-lab-kit` prompt optimization into Toolset-Training means more than adding prompt examples or one-shot rewrites. The target is a config-driven evolutionary tuner service: given a model, evaluation set, prompt subjects, operators, and stopping rules in YAML, generate an initial prompt population, evaluate candidates through existing Toolset evaluation surfaces, preserve the best prompts, mutate/crossover them, and repeat until the configured threshold or stop rule is met.

The current Toolset scaffold should remain as schema v1 one-shot optimization, but lab-kit parity requires a schema v2 evolutionary engine with population state, lineage, generation history, evaluator adapters, and SynthChat-compatible overlays.

#### Design Approach

Keep `PromptOptimizationService.from_config(...).run()` as the public API, but make it a facade over schema-aware internals.

Core runtime concepts:

- `PromptOptimizationConfig`: normalized validated YAML config.
- `PromptSubjectResolver`: current dotted-path subject loading plus required placeholder/constraint metadata.
- `PromptGenome`: one candidate containing all prompt subject values for a run.
- `PopulationBuilder`: creates generation 0 from baseline, configured catalogs, deterministic operators, and optional `llm_rewrite`.
- `OperatorRegistry`: applies mutation and crossover operators from config.
- `EvolutionLoop`: evaluates population, ranks candidates, preserves elites, selects parents, creates offspring, and applies stopping rules.
- `EvaluationAdapter`: normalizes fixture, Evaluator, SynthChat, or future RAG scoring into `score`, `metrics`, `passed`, `diagnostics`.
- `ArtifactWriter`: writes v1-compatible overlays plus v2 generation, lineage, replay, and report artifacts.

Data flow:

1. Load config and prompt subjects.
2. Build generation 0 population.
3. Evaluate every genome through the configured adapter.
4. Rank by configured objective metric.
5. Write generation history and candidate lineage.
6. Stop if `target_score`, `max_generations`, or `stagnation` rules trigger.
7. Otherwise preserve elites, select parents, mutate/crossover, and repeat.
8. Emit `overlays.json` for SynthChat plus v2 evolutionary artifacts.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Compatibility mode | Replace current service vs keep schema v1 | Keep schema v1 as one-shot mode | Avoid breaking existing configs/tests while v2 is added. |
| Candidate shape | Per-subject candidate vs multi-subject genome | Multi-subject genome for v2 | Lab-kit RAG evolves prompt sets together; per-subject winners can be incoherent. |
| Evaluation integration | Hardcode Evaluator calls vs adapter | Adapter | Keeps prompt optimization generic and config-driven. |
| Optimized prompt persistence | Write source YAML vs overlays | Overlays only by default | Preserves reproducibility and avoids surprising source mutation. |
| Model use in tests | LM Studio in unit tests vs fake client | Fake clients in tests, manual LM Studio smoke | Keeps CI deterministic and avoids local service dependency. |

#### Interface Contracts

Service API remains:

```python
result = PromptOptimizationService.from_config(path, overrides=None).run()
```

Normalized evaluator result:

```python
{
    "score": 0.0,
    "metrics": {},
    "passed": False,
    "diagnostics": [],
    "raw_artifacts": {}
}
```

V2 candidate history row:

```json
{
  "candidate_id": "...",
  "generation": 2,
  "genome": {"subject_id": "prompt text"},
  "parents": ["candidate-a", "candidate-b"],
  "operator": "crossover_subject_mix",
  "score": 0.82,
  "metrics": {},
  "selected": true,
  "diagnostics": []
}
```

`overlays.json` must remain compatible with the current SynthChat hook:

```json
{
  "version": 2,
  "selected_candidate_id": "...",
  "selected_candidate_ids": {"subject_id": "..."},
  "subjects": {
    "subject_id": {
      "source_path": "...",

      "source_path_repo_relative": "...",
      "dotted_path": "...",
      "selected_candidate_id": "...",

  }
}
```

## Current Decisions And Historical Questions

Evaluator-backed scoring; v2 optimizes multi-subject genomes; objective metrics
are config-selected with `stats.normalized_score` as the adapter default and
`stats.pass_rate` used by the lab-kit epistemic smoke; Evaluator failures use
`score_floor` by default with an explicit `raise` policy available. The
remaining bullets are preserved as the original planning record, not as current
blockers.
