# Implementation Plan: SynthChat Privacy Seed Sanitization

> Generated by Codex on 2026-04-23
> Status: PENDING APPROVAL

## Summary

Add a config-driven privacy preprocessing stage to SynthChat that can sanitize seed inputs before generation begins. The primary path is:

1. detect and mask PII/secrets locally with OpenAI Privacy Filter (`openai/privacy-filter`)
2. replace typed masks with realistic synthetic values using programmatic generators by default
3. optionally run an LLM polishing pass on already-sanitized text
4. feed the resulting sanitized seed text into normal SynthChat generation

This should be implemented as a reusable seed/data preprocessing layer, not as scenario prompt logic and not as a global dependency upgrade of the main training environment. Scenario or run config should opt into a named preprocess profile; the runtime then applies the profile before raw seed content reaches remote LLMs.

The design goal is to support both privacy-preserving masking and realistic pseudonymization while preserving the repo's config-first discipline.

#### Design Approach

Use a dedicated privacy preprocess pipeline with three separable layers:

1. **Detection/Masking Layer**
- local-only OPF wrapper
- input text in, detected spans and masked text out
- typed placeholders preferred internally, even if final output mode is generic redaction

2. **Replacement Layer**
- default provider: programmatic pseudonymization
- optional provider: custom generators for secrets/account numbers
- optional provider: LLM polish pass after programmatic replacement
- deterministic mapping for consistency within a configured scope

3. **Integration Layer**
- seed ingress hooks for docs and existing JSONL
- optional egress hook for generated examples before write
- metadata and labels describing what was changed

This keeps privacy transformation independent from scenario prompt wording while still letting scenarios opt into named behavior through config.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| OPF integration path | A) upgrade shared Transformers B) call HF pipeline directly C) use isolated `opf` runtime | **C** | Avoids destabilizing the training env and matches upstream runtime |
| Replacement default | A) LLM only B) programmatic only C) hybrid | **C** | Programmatic first for safety and determinism, optional LLM polish for fluency |
| Config attachment point | A) per-scenario prompt text B) named preprocess profile C) hardcoded runtime flag | **B** | Fits repo's config-first discipline |
| Consistency scope | A) random every occurrence B) document-level C) dataset-level configurable | **C** | Different use cases need different tradeoffs |
| Secret replacement | A) Faker B) custom generators C) leave masked | **B/C configurable** | Faker is weak for tokens/keys; masking may be safer |
| Raw PII visibility to LLM | A) allow for replacement B) never allow | **B** | The privacy layer should protect upstream calls, not depend on them |

#### Proposed Config Shape

Global defaults in `settings.yaml`:

```yaml
privacy_preprocess:
  enabled: false
  profile: null
  apply_to:
    docs: true
    input_jsonl: false
    generated_output: false
  on_error: fail   # fail | skip | keep_masked
```

Named profiles in a new config file, for example `SynthChat/config/privacy_profiles.yaml`:

```yaml
profiles:
  mask_only:
    detector:
      provider: opf
      checkpoint: null
      device: cpu
      output_mode: typed
    transform:
      mode: mask
      keep_typed_placeholders: true

  realistic_pseudonyms:
    detector:
      provider: opf
      checkpoint: null
      device: cpu
      output_mode: typed
    transform:
      mode: pseudonymize
      provider: hybrid
      consistency_scope: document
      faker_locale: en_US
      fake_email_domain: example.com
      preserve_date_shape: true
      preserve_account_number_shape: true
      secret_strategy: mask
      llm_polish:
        enabled: false
```

Scenario or target reference:

```yaml
seed_data:
  preprocess_profile: realistic_pseudonyms
```

This keeps the scenario interface simple while moving the actual privacy policy into a reusable config registry.

#### Interface Contracts
- Privacy preprocessing runs before seed content is interpolated into scenario prompts.
- When docs are used, `doc_content` passed into templates is already sanitized if a profile is enabled.
- When JSONL examples are ingested, sanitized content must preserve overall example structure.
- The replacement layer must expose deterministic mappings per configured scope.
- Generated artifacts should record whether privacy preprocess ran, what profile was used, and what labels/types were detected.
- No code path should require raw PII to be sent to a remote LLM in order to complete sanitization.

#### Implementation Sequence

1. **Config foundation**
- add `privacy_preprocess` to `settings.yaml`
- add `privacy_profiles.yaml`
- add loader/resolution helpers

2. **Local runtime wrapper**
- implement `privacy_filter.py` around `opf`
- support checkpoint path, device, output mode, and thread/process-local cache

3. **Programmatic replacement engine**
- implement `pseudonymizer.py`
- add deterministic name/email/phone/address/url/date generators
- add custom generators or mask-only behavior for secrets/account numbers

4. **Preprocess orchestrator**
- implement `privacy_preprocess.py`
- wire detect -> replace -> optional polish -> metadata summary

5. **Seed ingress integration**
- docs path
- JSONL input path
- generator path before prompt rendering

6. **Optional output sanitization**
- apply to generated examples before write when configured
- attach metadata + labels

7. **Standalone sanitize command**
- allow dataset sanitization independent of generation/improvement

#### Runtime Isolation Strategy

Do **not** add Privacy Filter support by upgrading the main training stack. Instead:

- keep main repo dependencies unchanged
- install `opf` into an isolated helper environment or helper path
- invoke it through a thin Python API wrapper when available
- fail clearly if a privacy profile is enabled but the helper runtime is missing

This mirrors repo guidance to avoid destabilizing existing cloud/training stacks for unrelated runtime needs.

#### Caching Requirement

The privacy runtime must be cached across repeated operations. Parallel generation currently constructs fresh generators per work item, so the privacy wrapper needs its own thread-local or process-local model cache. Reloading a 1.5B model per example is out of scope and would make the feature unusable.

#### Executable Smoke Dataset

Create a checked-in fake fixture set for privacy smoke testing, for example:

- `tests/fixtures/privacy/raw_seed_docs/customer_support_email.txt`
- `tests/fixtures/privacy/raw_seed_docs/crm_export.md`
- `tests/fixtures/privacy/raw_seed_docs/dev_notes_with_secrets.txt`
- `tests/fixtures/privacy/raw_seed_dataset.jsonl`

These fixtures should contain **synthetic but realistic** PII-like values only:

- fake names
- fake personal emails
- fake phone numbers
- fake mailing addresses
- fake dates of birth / appointment dates
- fake URLs
- fake account-like strings
- fake passwords / API-key-like tokens

The point is not sensitive-data protection in the test corpus; the point is verifying that the pipeline detects, masks, and replaces realistic patterns correctly.

#### Executable Smoke Workflow

Phase 1 should include a runnable local smoke workflow, not just unit tests:

1. create synthetic fixture docs and JSONL rows under `tests/fixtures/privacy/`
2. run standalone sanitize with the `mask_only` profile on the fixture corpus
3. inspect emitted spans, masked output, and metadata
4. run standalone sanitize with the `realistic_pseudonyms` profile
5. confirm replacements are realistic and consistent within the configured scope
6. run a small SynthChat docs-based dry run using the sanitized fixture docs as `--docs`
7. verify that rendered prompts and outputs do not contain the original fake PII strings
8. if `llm_polish` is enabled, run the polish step against a local model endpoint and compare before/after outputs

#### vLLM-Backed Polish Smoke

For the optional `llm_polish` path, the rollout should explicitly test against a local OpenAI-compatible serving stack such as vLLM.

Recommended shape:

- start a local vLLM OpenAI-compatible endpoint on `VLLM_HOST` / `VLLM_PORT`
- point the polish client at that endpoint
- feed it only already-sanitized text
- compare:
  - masked text
  - pseudonymized text
  - pseudonymized + polished text

Success criteria for the vLLM smoke:

- the model receives no raw PII
- it preserves placeholder replacements rather than reintroducing source values
- it improves fluency only around the replaced entities
- it does not alter high-risk fields such as masked secrets when those are configured to remain masked

Because the repo does not currently expose an explicit `vllm` or generic custom OpenAI-compatible provider in `shared.llm`, phase 1 should either:

- add a minimal custom OpenAI-compatible provider for polish-only use, or
- temporarily route the polish call through the existing LM Studio-style OpenAI-compatible client pointed at the vLLM host/port

Either way, the plan should treat vLLM as an actual runnable smoke target, not as a future-only idea.

#### Dry-Run Requirement

Before broad rollout:

1. enable `mask_only` on a small docs-based SynthChat dry run
2. inspect `doc_content`, metadata, and generated prompts
3. enable `realistic_pseudonyms` on a 3-5 example dry run
4. confirm no raw source PII appears in prompts or outputs
5. only then enable the feature for larger runs

This follows the repo's SynthChat dry-run discipline.

## Recommended Default Behavior

The initial recommended production default is:

- OPF detection enabled only when a profile is selected
- typed masking internally
- programmatic pseudonymization for people/contact/address/date/url data
- secrets left masked, not faked
- account-number-like data either left masked or replaced with shape-preserving synthetic values only when explicitly configured
- no LLM polishing enabled by default

This gives the repo a safe, testable baseline before introducing optional fluency polishing.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| OPF runtime is heavy or slow on CPU-only hosts | Medium | High | Make feature opt-in, cache runtime aggressively, start with small dry runs |
| Programmatic replacements break document coherence | Medium | Medium | Use deterministic mapping tables and optional post-sanitize polish |
| Faker-generated outputs are unrealistic for certain domains | Medium | Medium | Support profile-specific generators and domain-aware overrides |
| Secrets/account numbers are replaced unsafely | Medium | High | Default to masking for high-risk labels |
| Raw seed content still leaks to prompts through an unhooked path | Low | High | Add ingress tests for docs, JSONL, and generator prompt rendering |
| Shared environment destabilization from dependency changes | Low | High | Keep OPF in isolated helper runtime |
| Model reload cost in parallel mode is unacceptable | High | High | Require runtime cache before rollout |

## Scope

This plan covers privacy preprocessing for SynthChat seed and dataset inputs, plus optional output sanitization metadata. It does not attempt to:

- redesign scenario prompt templates around privacy logic
- replace OPF with a custom detector
- solve full compliance/anonymization certification
- fine-tune Privacy Filter in phase 1

Phase 1 should deliver a safe, config-driven, local-first privacy preprocess layer that can be turned on per profile and verified with small SynthChat dry runs.
